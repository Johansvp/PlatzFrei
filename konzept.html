PlatzFrei · Backend-Konzept

Das System hinter der App

Wie PlatzFrei als echtes, serverbasiertes Produkt aufgebaut wäre — Architektur, Datenmodell und die wichtigsten Konzepte.

1 · Ausgangslage

Prototyp vs. echtes System

Der vorliegende Prototyp läuft vollständig im Browser: Konten, Buchungen, Freundschaften, Gruppen und Nachrichten liegen im localStorage des jeweiligen Geräts. Das genügt, um alle Abläufe erlebbar zu machen — Daten werden jedoch nicht geteilt und gehen beim Löschen des Browser-Caches verloren.

Ein echtes Produkt verlagert die Daten auf einen Server mit zentraler Datenbank. Erst dadurch sehen alle Nutzer dieselben Belegungen in Echtzeit, Konten gelten geräteübergreifend, Gruppen-Chats synchronisieren sich live und Buchungen sind verlässlich gegen Doppelbelegung gesichert.

Kernidee: Der App-Aufbau (Screens, Abläufe, Funktionen) bleibt identisch. Ausgetauscht wird nur die Datenquelle — vom lokalen Speicher hin zu einer API mit zentraler Datenbank.

2 · Systemarchitektur

Wie die Teile zusammenspielen

Drei Schichten: die Clients (App & Website), das Backend (API-Server mit der Logik) und die Datenbank. Die Clients sprechen über HTTPS mit der API; nur die API darf in die Datenbank schreiben. Gruppen-Chat und Live-Belegung laufen zusätzlich über einen Echtzeit-Kanal.

5 · Schlüssel-Konzepte

Worauf es technisch ankommt

Keine Doppelbuchung

Vor dem Speichern prüft der Server in einer Transaktion, ob sich die Zeit mit einer bestehenden Buchung überschneidet. Zusätzlich verhindert eine Datenbank-Regel (Ausschluss überlappender Zeiträume je Standort) Doppelbelegung auch bei gleichzeitigen Anfragen.

Sicherheit & Login

Passwörter werden nie im Klartext gespeichert, sondern gehasht (z. B. bcrypt). Nach dem Login bekommt die App ein Token (JWT), das sie bei jeder Anfrage mitschickt. Alles läuft über HTTPS — DSGVO-konform. Die Nutzersuche gibt keine Passwörter oder E-Mails zurück.

Echtzeit-Belegung & Chat

Über einen WebSocket meldet der Server neue Buchungen und Gruppen-Nachrichten sofort an alle offenen Apps. Ein gebuchter Slot verschwindet in Echtzeit auf allen Geräten — ebenso erscheinen Gruppen-Chats live ohne Neuladen.

Absage-Benachrichtigung

Storniert ein Nutzer eine Buchung, sendet der Server automatisch eine Absage-Nachricht an alle eingeladenen Freunde (1:1-Chat) und Gruppen (Gruppen-Chat). Die buchung_einladung-Tabelle stellt sicher, dass keine Absage doppelt gesendet wird.

Gruppen & Gruppeneinladungen

Gruppen haben eigene Chat-Verläufe in der Datenbank. Bei einer Buchung können ganze Gruppen eingeladen werden — der Server schickt die Einladungs-Nachricht automatisch in den jeweiligen Gruppen-Chat, ohne dass der Nutzer jeden einzeln anschreiben muss.

Profilbilder

Bilder landen in einem Objektspeicher (statt direkt in der Datenbank); gespeichert wird nur die Adresse (profilbild_url). Das hält die Datenbank schlank. Im Prototyp werden Bilder als Base64 im localStorage gespeichert — das skaliert nicht für viele Nutzer.

7 · Der Weg dahin

Vom Prototyp zum echten Backend

Fazit: Der Prototyp zeigt das vollständige Verhalten — Buchen, Stornieren, Gruppen, Chat, Freunde, Profil — dieses Konzept zeigt das Fundament dahinter. Zusammen ergibt sich ein vollständiges Bild von PlatzFrei als marktfähigem Produkt.
